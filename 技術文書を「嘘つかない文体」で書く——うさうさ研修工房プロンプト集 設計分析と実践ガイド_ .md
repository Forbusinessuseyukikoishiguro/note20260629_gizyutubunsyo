# はじめに

「技術文書が書けない」という悩みの本質は、書き方の問題より**設計の問題**であることが多い。  
何を書くべきか決まっていない状態で書き始めるから、文章が散漫になる。

この記事では、うさうさ研修工房（石黒友季子）が構築した**技術文書生成プロンプト集 v1.0**を設計の観点から分析し、「なぜこの構造になっているのか」を解剖する。プロンプトをコピペするだけでなく、**設計思想を理解して自分用にカスタマイズできる**ことを目標とする。

# 第1章　設計分析——このプロンプト集が解いている問題

## 1-1. 解いている問題は何か

技術文書を Claude に書かせると、よくある失敗パターンが3つある。

1. **根拠なき断言問題**：「〜が最適です」「〜が正解です」という断定が混入する  
2. **構成ブレ問題**：依頼するたびに章立てが変わり、複数文書を並べると統一感がない  
3. **読者不在問題**：「書いた人が分かる」文書になり、「読む人が分かる」文書にならない

このプロンプト集は、6本のプロンプトと詳細な制約仕様によってこの3問題を構造的に封じている。

## 1-2. 設計の4本柱

### 柱1：構成の固定化（PROMPT-01）

章立てを「第0章〜第6章＋付録」と固定し、章番号を漢数字で統一する。

第0章　概要（一言・設計思想・動作環境）

第1章　要件定義書（FR / NFR テーブル）

第2章　詳細設計書（アーキテクチャ・モジュール）

第3章　実装仕様書

第4章　デバッグ方法

第5章　カスタマイズ方法

第6章　ソースコード解説

付録　用語ミニ辞典（6語以内）

**設計の肝**：第0章を先頭に置くことで、読者は「何のツールか」を把握してから詳細に進める。要約を末尾に置く日本語文書の慣習をあえて破っている。

### 柱2：文体の明文化（「嘘つかない原則」）

\- 体言止め禁止

\- 受動態を最小化

\- 主語を省略しない

\- 1文は60字以内

\- 根拠なき断言をしない

\- 推測には「〜の可能性」を付ける

\- 未検証・例外・制約は「\*\*重要\*\*：」ラベルで明示

文体制約は「何を書くか」ではなく「どう書かないか」を規定する。禁止事項を明確にすることで、AI の出力にブレが生まれにくくなる。

### 柱3：ID 付きテーブルによる参照可能性

要件テーブルには必ず `FR-01`、`NFR-01` のような ID を付ける。  
これにより、「FR-07参照」と本文から指定参照ができ、文書がコードと同様に**保守可能**になる。

### 柱4：コードブロックの標準化

コードの前：「何のための関数か」を1文で宣言

コードの中：行末に ① ② ③ のステップ番号コメント

コードの後：「\*\*設計の肝\*\*：〜」または「\*\*ポイント\*\*：〜」で締める

1ブロック20〜30行以内

「コードを置く」のではなく「コードを語る」構造になっている。

# 第2章　各プロンプトの解剖

## PROMPT-01：技術仕様書をゼロから書かせる

**特徴**：最も詳細な制約を持つマスタープロンプト。他のプロンプトはすべてここから派生している。

**肝となる制約**：動作環境をテーブルで書かせる指定。ここを自然言語で書かせると「最新ブラウザで動作します」のような曖昧な記述が生まれる。テーブル強制によって `対応ブラウザ | Chrome 120+ / Firefox 121+ / Safari 17+` のような具体的な記述が得られる。

**素材欄の重要性**：末尾の「素材（あれば貼る）」セクションに、既存 README やソースコードを貼ることで、AI は「推測」ではなく「根拠ある記述」ができるようになる。素材なしで使うと、根拠のない断言が混入しやすい。

---

## PROMPT-02：ソースコードから仕様書を逆引きする

**特徴**：「生成手順」を明示している点が他のプロンプトと異なる。

1\. コード全体を読み、「何をするツールか」を50字で言語化する

2\. 設計思想を4本柱で抽出する

3\. 実装済み機能を FR テーブルに整理する

4\. モジュール・関数を層ごとに分類する

5\. 主要関数に ① ② ③ コメントを付けた解説を書く

AI に手順を踏ませることで、「コードを読んで要約する」という漠然とした依頼より精度が上がる。**Chain-of-Thought の仕様書版**と言える設計になっている。

:::message alert PROMPT-02 の「任意」章の選択は重要。全章出力させると長大になる。デバッグ章（第4章）が必要かどうかは、対象が内部利用ツールか公開ツールかで変わる。 :::

---

## PROMPT-03：Zenn 記事を仕様書から生成する

**特徴**：「生成しないもの」セクションがある。

\#\# 生成しないもの

\- 著者自己紹介・SNS リンク・アフィリエイト

\- 体験談口調（「やってみた」「感動した」等）

\- 根拠のないベストプラクティス断言

\- セクション外の前置き・後記

「何を書くか」だけでなく「何を書かないか」を明示することで、Zenn の技術記事によく見られる冗長な前置きや根拠のない断言を構造的に排除している。

**Zenn 記法の活用**：`:::message` / `:::details` の使用を積極的に指示している。これは単なる装飾ではなく、「補足情報とメイン情報の分離」という情報設計上の判断を反映している。

---

## PROMPT-04：部分生成（A〜E）

PROMPT-04 は5つのサブプロンプトの集合体。用途別に切り出して使うためのモジュール設計になっている。

| サブ | 用途 | 特記 |
| :---- | :---- | :---- |
| 04-A | 第0章だけ | 一言説明（50字以内）が品質の試金石になる |
| 04-B | FR / NFR テーブルだけ | 機能リストを渡すと自動振り分けする |
| 04-C | ソースコード解説だけ | ① ② ③ の番号コメントが中核 |
| 04-D | デバッグ章だけ | 症状列は「ユーザーが観察できること」に限定 |
| 04-E | カスタマイズ章だけ | 末尾に「機能を足すときの作法」節を必ず加える |

---

## PROMPT-05：レビュー依頼

**特徴**：評価記号を `✅ / ⚠️ / ❌` の3段階に限定している。合否の二値ではなく「改善余地あり（⚠️）」を設けることで、完璧でなくても前進できる設計になっている。

1\. 評価テーブル（✅ / ⚠️ / ❌）

2\. ❌ の箇所：問題のある文と修正案を並べて示す

3\. ⚠️ の箇所：改善の余地がある点を1文で指摘

4\. 総評（3文以内）

「修正案を並べて示す」という出力形式の指定が重要。「問題点だけ指摘する」レビューでは、書き手が修正方法に迷う。

# 第5章　カスタマイズガイド

## 5-1. 自分のスタイルに合わせる

「うさうさ研修工房スタイル」を自分用に変えるときは、PROMPT-01 の「文体ルール」セクションを書き換える。変更してよい箇所と注意が必要な箇所を以下に示す。

| 変更箇所 | 変更内容の例 | 注意点 |
| :---- | :---- | :---- |
| 章立て | 章を増減する | 章番号の参照が文書内で壊れる可能性あり |
| 文字制限 | 1文80字以内に変更する | 制限を緩めると AI の文が長くなりやすい |
| ID 形式 | `FR-` を `REQ-` に変更する | 文書間の参照で混在に注意する |
| 評価記号 | `✅ ⚠️ ❌` を `OK / WARN / NG` に変更する | ASCII のほうが検索しやすい場合がある |

## 5-2. 対象読者を変えるとき

PROMPT-03 の「対象読者」行を書き換えることで、記事のトーンが変わる。

\# 変更前

\- 対象読者：JavaScript / Web エンジニア（中級）

\# 変更例1（新人向け）

\- 対象読者：プログラミング学習中の初学者。専門用語には必ず括弧で日本語の説明を付ける

\# 変更例2（上級者向け）

\- 対象読者：シニアエンジニア。設計判断の根拠と代替案の比較を重視する

## 5-3. 仕様書なしで Zenn 記事を書かせるとき

PROMPT-03 の末尾「\#\# 仕様書」セクションを「\#\# トピック情報」に変えて、以下を貼る。

\#\# トピック情報

\- テーマ：（記事のテーマ）

\- 書きたい実装の要所：（3〜5点を箇条書きで）

\- 対象コードの言語：（js / ts / python 等）

\- 想定する読者のレベル：（初級 / 中級 / 上級）

# おわりに

このプロンプト集の本質は「AI に仕様書を書かせる」ではなく、「**書き手が何を決めるべきかを明示する**」ことにある。

プロンプトは、書き手の思考の型を外部化したものだ。「体言止め禁止」も「ID 付きテーブル」も「嘘つかない原則」も、元は書き手が頭の中で持っていたルールを文字にしたに過ぎない。

AI が普及した時代の技術文書の品質は、**プロンプトの精度ではなく、書き手の設計思想の解像度**によって決まる。
